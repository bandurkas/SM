# MemPalace Session Diary: 2026-04-23

## 🛡️ Wing: void_chromosphere (VPS/ethbot)
## 📝 Session Summary: Auto-Heal State Desync Bugfix

### 1. The Incident
- **Event**: User manually opened a tiny short position (0.03 ETH) directly in the HTX app while the bot was holding a live LONG of 0.81 ETH.
- **Root Cause**: Because the HTX account uses One-Way Mode (Односторонний режим), manually opening a short automatically forced the exchange to close the bot's Long position. The bot's internal `state.json` became desynchronized, still thinking it was in a Long, and continued moving an imaginary trailing stop.

### 2. The Fix (Auto-Healing State)
- **Problem**: `reconcile_with_exchange()` was previously only called *once* when the bot started up.
- **Solution**: Injected `manager.reconcile_with_exchange()` directly into the main `while True` loop of `eth_bot.py`.
- **Result**: The bot now queries the actual exchange positions on every single bar (every 15 mins). If it detects that a human manually closed a trade (or opened a reverse trade), the bot instantly recognizes the mismatch (`WARNING State/exchange side mismatch`), drops its active state, and safely resets to a neutral monitoring mode without requiring a manual reboot.

### 3. Critical Fix: Ghost Trade Bug (Auto-Adoption)
- **Problem**: Moving `reconcile_with_exchange()` to the main loop exposed a race condition. If a pending limit order filled on the exchange *before* the 15-minute candle closed, the bot's paper-logic hadn't yet registered the fill. When the auto-heal check ran, it saw a live position but no "state" trade, branded it an "ORPHAN", and deleted its own pending order, leaving the live trade unprotected.
- **Solution**: Refactored `trade_manager.py` to intelligently *auto-adopt* the orphan. If `live_contracts > 0` and it matches the `pending.side`, the bot immediately transforms the pending order into a live `ActiveTrade` and deploys the Stop-Loss to the exchange.
- **Status**: Code patched, pushed, and bot restarted.

# ⚠️ POST-MORTEM & INCIDENT REPORT: 2026-04-23
**Subject**: The "Ghost Trade" Orphaned Long & State Desync
**Impact**: -$53 Unrealized PNL (14.7% Drawdown on 10x Margin)

### 1. What Happened
Between April 22 19:30 UTC and April 23 04:00 UTC, the bot successfully placed a LONG limit order for 0.90 ETH at `2397.09`. The order filled on the exchange. However, due to a state tracking bug, the bot assumed the order had been manually cancelled, deleted the trade from its memory (`state.json`), and failed to place a Stop-Loss. 

The market subsequently dumped from ~2390 to 2336. The unprotected 0.90 ETH LONG absorbed the full impact of the dump, tying up margin and preventing the bot from catching the actual local bottom at 2336 (resulting in an `InsufficientFunds` error).

### 2. Root Cause Analysis
- **Race Condition in `trade_manager.py`**: The bot verifies pending limit orders using *paper trading logic* (checking if the OHLCV candle `low` crossed the limit price).
- If the exchange fills the order immediately (e.g. due to spread) *before* the 15-minute candle closes, the order disappears from `fetch_open_orders()`.
- The bot saw the order disappear from the exchange, but its paper logic said it hadn't filled yet. The bot concluded: *"The user must have cancelled it in the app"*.
